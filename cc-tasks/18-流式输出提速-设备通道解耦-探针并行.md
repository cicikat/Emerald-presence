# CC 任务 18：流式输出提速 —— 打点、设备通道解耦、探针并行

> 背景：茶茶反馈"输出速度比以前慢了"，时间点与 ESP32 显示屏接入吻合。实读代码后确认存在真实机制（问题 B），不全是错觉；同时补打点让以后的"感觉慢"可以直接看数字。
> 三项独立，按 A → B → C 顺序做。

## A · 每轮耗时打点（先做，验证 B/C 的标尺）

`admin/routers/chat.py` `run_owner_chat_turn` 里记录并在轮末 INFO 一行输出：

```
[owner_chat/timing] probe=1.82s ctx=0.31s prompt=0.02s first_delta=2.45s stream=8.10s chars=812 (100.2 c/s) post=1.95s total=14.7s
```

- `probe`：`_probe_and_execute_tools` 耗时；`ctx`：fetch_context；`first_delta`：run_llm_stream 首个 piece 距发起；`stream`：首 delta 到 stream_end；`chars/c/s`：可见字速；`post`：record_assistant_turn 耗时。
- 非流式分支打 `llm=` 代替 first_delta/stream。
- 纯 logging，不改任何行为。**做完 A 先跑两条对照：设备连接 vs 断开，字速差异直接定位问题 B 是否命中。**

## B · 设备通道解耦 token 循环（主要嫌疑）

**现状**：`chat.py` 流式循环里 `await _ui_push.push_stream_delta(...)`，而 `ui_push.push_stream_delta` 对桌面、设备**逐 token 串行 await `send_text`**。ESP32 走 WiFi 且渲染时不读 socket，TCP 缓冲一满 send 就阻塞——**整条流被最慢客户端限速**，桌面端跟着慢。

**修法**（改 `channels/device_ws.py` + `channels/ui_push.py`，desktop_ws 不动）：

1. `device_ws` 内建**出站队列 + 单 writer 任务**：
   - `_out_queue: asyncio.Queue`（maxsize ~64）随连接建立创建、断开时取消 writer 并清空。
   - 新增 `enqueue_json(payload) -> bool`：非阻塞入队；队满时执行**delta 合并**——若队尾与新帧都是 `message_stream_delta` 且同 msg_id，把 delta 字符串拼接合并为一帧（无损）；其他类型帧队满则丢弃并 WARN（stream_start/end、channel_message、segments 不该在 64 深度下丢，真丢了说明设备已死，心跳会踢）。
   - writer 任务：循环 `get()` → `send_text`，发送异常时断开清理（复用现有断开路径）。
   - 可选节流：writer 里对连续 delta 帧做 ~100ms 聚合（收到 delta 后 sleep 0.1 再把队列里后续同 id delta 一并合并发出），设备屏幕不需要逐 token 刷新。
2. `ui_push.push_stream_delta / push_stream_start / push_stream_end / push_segments`：设备分支改为 `device_ws.enqueue_json(...)`（同步、不 await 网络）；桌面分支保持直接 await（本机回环，快且保序习惯不变）。
3. `push_message`（channel_message）与 action ack 路径**不改**：channel_message 走队列即可（enqueue），`push_action_and_wait` 保持直发等 ack 的现有逻辑。
4. 固件侧不需要改（帧格式不变，只是 delta 可能合并变大——固件解析本就按帧处理 delta 字符串，合并无影响；如固件对单帧长度有上限，writer 聚合上限设 512 字节）。

## C · 探针与 fetch_context 并行（顺手的几百毫秒）

`run_owner_chat_turn` 里两者串行但互不依赖（探针自读 short_term 最近 4 条，不吃 context）：

```python
tool_result_text, context = await asyncio.gather(
    _probe_and_execute_tools(_probe_text, user_id, char_id=_frozen_scope.character_id),
    pipeline.fetch_context(user_id, message, frozen_scope=_frozen_scope),
)
```

- 注意两处都在 conversation_lock 内，gather 不改变锁语义。
- 探针含**副作用工具**（desktop 动作等）：现状也是在 fetch_context 之前执行完毕才建 prompt，并行后语义不变（工具结果仍在 build_prompt 前就绪）。唯一变化是工具副作用与 fetch_context 的磁盘读并发——工具不写记忆层，无冲突；如有疑虑，给 memory 类工具维持现状（该类本就未暴露）。

## 不做 / 说明

- **9k prompt + 探针的基线延迟属设计成本**：探针是一次完整 LLM 往返，决定首字前的等待大头；C 只能并行掉 fetch_context 那截。若将来要进一步压首字延迟，方向是给 desktop 路径复用 QQ 侧的关键词快路径（命中明确关键词才跳过 LLM 探针）——本单不做，先看 A 的数字再定。
- 手机端流式化是 14 号单的范围，不在此单。

## 验证

- A：日志出现 timing 行，字段齐全；设备断开 vs 连接各发一条对照。
- B：设备连接状态下桌面字速与设备断开时基本一致（差 <10%）；设备端仍能正常显示流式/整段消息；拔设备电源（不优雅断开）后发消息，桌面不受影响，心跳 70s 内踢掉死连接。
- C：timing 里 `probe` 与 `ctx` 不再串行相加（total 减少约 min(probe,ctx)）。
