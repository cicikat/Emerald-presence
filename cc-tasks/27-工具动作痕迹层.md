# Brief 27 · 工具动作痕迹层(action trace)

> 问题:工具结果只在**执行当轮**注入 prompt(层 `10_tool_result`),下一轮角色就"失忆"——
> 用户追问"你刚才查到什么/你干了什么"无从溯源,探针路径尤其明显。
> 方案:每次工具执行落一条精简痕迹,注入新 prompt 层"你最近做过的操作",并可选写 event_log
> 让动作进入记忆固化链。这是后续 tool loop(Brief 待定)的前置件,但独立可用。

---

## 1. 现状盘点(已确认)

- 工具执行唯一收口:`core/tool_dispatcher.py:963 execute()`,返回 `(tool_result, ask_confirm_text)`,origin 白名单 `user_live`(路径A探针)/ `assistant_intent`(路径B意图)。**两条路径都过这里**,是理想的单点埋点位。
- 结果注入:`core/prompt_builder.py:1217`,有 `tool_result` 时经 `to_tool_result/frame_tool_result` 生成层 `10_tool_result`;层清单表在 ~1635。**仅当轮有效,无跨轮存留。**
- 已有先例 `core/memory/tool_read_log.py`(P2 已读指纹):证明"工具侧写 per-uid 小文件"这个形态已被接受,但它只做去重,不做展示。
- `event_log.append`(`core/memory/event_log.py:206`)支持 `trigger_name` meta 字段,可标记来源。

---

## 2. 方案

### 2.1 存储:`core/memory/action_trace.py`(新文件)

- 路径:参照 `event_log` 的接法,`MemoryScope.reality_scope(uid, char_id)` + `resolve_path` 得到
  `data/runtime/memory/{char_id}/{uid}/action_trace.json`。**不得硬编码 data/ 路径**(硬性规则1);
  若 resolve_path 需要注册新 key,照 event_log 的注册方式补。
- 结构:JSON 数组,环形上限 **30 条**,超出裁头。原子写(`core/safe_write.py`)。
- 单条 schema:

```json
{
  "ts": 1789000000.0,
  "tool": "web_search",
  "origin": "user_live",
  "args_digest": "query=明天北京天气",
  "result_digest": "北京明天多云,18-26度…",
  "status": "ok"
}
```

- `args_digest`:从 tool_args 提取**白名单字段**拼接(每工具在 `_TOOL_REGISTRY` 条目可选新增 `trace_args: [...]` 字段声明;未声明的工具只记工具名,不记参数——防 secrets/长文本入痕迹)。截断 60 字。
- `result_digest`:取 `to_tool_result().safe_summary` 前 80 字(复用现有脱敏出口,**不碰原始 result**,尤其 `peek_screen_content` 的受控出口约束不被绕过——痕迹里它只记"看了一眼屏幕:{title_hint}",不记 visible_text)。
- `status`:`ok` / `failed` / `pending_confirm`(ask_confirm_text 非空时)。失败也记,角色下轮才知道"刚才没成功"。
- API:`record(uid, char_id, entry)` / `recent(uid, char_id, *, max_items, window_hours) -> list`。全 fail-open(记录失败只 log,不影响工具主流程)。

### 2.2 埋点:`execute()` 收口处

`tool_dispatcher.execute()` 成功/失败/待确认返回前统一调 `action_trace.record(...)`。
注意:origin 闸门拒绝(fail-closed 那支)**不记**——那不是角色做过的事。

路径B(`pipeline._parse_and_execute_intent` → desktop action)若有不经 `execute()` 的直推分支
(`_push_desktop_action` 直调处),同样补记,status 依据 ack/降级结果。以代码实际为准,cc 执行时先读
`docs/tools.md` 路径B一节确认闸门顺序。

### 2.3 注入:新 prompt 层 `10.5_action_trace`

- `fetch_context()` 里读 `action_trace.recent(max_items=5, window_hours=24)`,传入 `prompt_builder.build(action_trace_entries=)`。
- `prompt_builder`:非空时紧跟 `10_tool_result` 之后注入,**必须带 `_layer: "10.5_action_trace"`**(硬性规则3),并登记进 ~1635 层清单表。
- 文案框架(第二人称,与层10口径一致):

```
（你最近做过的操作,供回忆,不要逐条复述:
- 14:02 查了天气:北京明天多云,18-26度…
- 13:40 打开了网页:xxx
）
```

- **当轮去重**:本轮已有 `tool_result` 时,跳过与之同源的最新一条(ts 在本轮开始之后且 tool 同名),避免层10/层10.5 说同一件事两遍。
- 预算:全层截断 400 字。**不进裁剪优先链**(event_search→mid_term→diary→episodic→lore 顺序不动),因为它够小且时效性强;但带 `_layer` 保证裁剪逻辑能看见它。

### 2.4 可选回流:event_log(默认开,可配)

`record()` 中 `status=ok` 且 config 开启时,追加一条:

```python
event_log.append(uid, "assistant", f"（做了一件事:{tool} — {result_digest[:40]}）",
                 trigger_name="action_trace", char_id=char_id)
```

作用:动作进入 event_log → 被角色日记(Brief 26)和 event_search 看见,"他今天帮她查了三次天气"
能沉淀成记忆。`trigger_name` meta 保证可溯源。
风险与开关:见 §4;config 键 `action_trace.event_log_echo: true`。

### 2.5 配置

`config.example.yaml` 新增(config.yaml 同步):

```yaml
action_trace:
  enabled: true
  inject_max_items: 5
  inject_window_hours: 24
  event_log_echo: true
```

`enabled: false` 时:不记录、不注入,零行为变化(回滚开关)。

### 2.6 文档同步

- `docs/tools.md`:新增"动作痕迹"一节(埋点位置、脱敏规则、trace_args 字段说明)。
- `docs/prompt-layers.md`:登记 `10.5_action_trace` 层(条件、预算、去重规则)。

## 3. 测试

新增 `tests/test_action_trace.py`:

1. `execute()` 成功 → 落一条 status=ok;工具抛错 → status=failed;ask_confirm → pending_confirm;origin 拒绝 → **不落**。
2. 环形上限:塞 35 条 → 文件只剩 30 条且是最新的。
3. `trace_args` 白名单:未声明工具的 args 不出现在 args_digest;声明字段截断 60 字。
4. 注入:有痕迹时层 `10.5_action_trace` 出现且带 `_layer`;空痕迹/enabled=false 时不出现。
5. 当轮去重:同轮 tool_result 存在时,最新同源条目不进层10.5。
6. event_log_echo:开→ event_log 出现 `trigger: action_trace` 行;关→ 不出现。
7. 敏感回归:`peek_screen_content` 的痕迹不含 visible_text/clickable_text 内容(只允许 title_hint)。
8. tag_rules 未动,无需跑 run_eval;但若 cc 实现中触碰了层激活逻辑,按硬性规则4跑 `python tests/run_eval.py`。

## 4. 风险与不做什么

- **风格污染**:event_log 里的动作行会进日记生成的对话日志。预期是加分项("今天帮她查了天气"),
  但若观察到日记大量复述动作清单,先关 `event_log_echo` 止血——所以它必须是独立开关。
- **prompt 预算**:20k 硬限下新增 ≤400 字,风险低;不重排现有裁剪顺序。
- 不做:tool loop / 多步执行(后续 brief);不给主 LLM 挂 tools schema;不改探针 prompt;
  不动 `_sanitize_assistant_message`(痕迹不写入 short_term history,只走独立层,正是为了绕开
  风格反馈坍缩风险)。
- 硬性规则6(provenance_log)针对 identity/episodic/mid_term/trait/author_note 写入点,
  action_trace 不在其列,不需要接;event_log_echo 走的是既有 append,自带 trigger meta。
