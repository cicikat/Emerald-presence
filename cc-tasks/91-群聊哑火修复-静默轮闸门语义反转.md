# Brief 91 · 群聊哑火修复：静默轮闸门语义反转（一行病根 + 回归网）

## 根因（已定位，不用再查）

`core/stage/runner.py:169-172`（commit `3a3b32e`，Brief 53 系引入）：

```python
may_be_silent = (
    not has_vocative and initial_ranked and initial_ranked[0].total < SILENCE_THRESHOLD
    and (is_low_information(owner_content) or stage.settings.allow_silent_rounds)
)
```

`allow_silent_rounds` 默认 **True**（`models.py:27`），`or` 把低信息闸门整个短路——
条件退化为「没点名 && 分数 < 0.35 就静默」。而普通陈述句（不点名、无问号、不踩
keywords）的裸分只有 `talkativeness 0.5 × 0.5 = 0.25`，再扣 recency penalty 更低，
**永远 < SILENCE_THRESHOLD(0.35)** → 每轮静默，`min_responders` 保底完全被绕过。
这就是「发话没有任何角色响应」。与私聊机制（Brief 86）无关——时间上挨着跑纯属巧合，
86 的改动只在 scheduler 侧和生成视图，不碰 Phase A 判定。

面板佐证路径：群聊仲裁页的仲裁轨迹里应该能看到连排的「静默轮次」badge、
top 分 0.1~0.25。

## 1. 🟡 修复（语义修正，不是调阈值）

Brief 53 的本意是「**低信息消息**（嗯/哦/表情）才允许无人接」。改为：

```python
and stage.settings.allow_silent_rounds and is_low_information(owner_content)
```

即：设置开 **且** 消息本身是 backchannel，才允许静默。有内容的消息永远走
`min_responders ≥ 1` 保底。**不调 SILENCE_THRESHOLD、不调打分权重**——打分体系
（85 已回测）没病，病在闸门连接词。

## 2. 🟢 顺手：静默轮 trace 补充原因字段

`_append_arbiter_trace` 的 `extra={"silent_round": True}` 加 `"silent_reason":
"low_information"`，面板已渲染 badge，加字段即可被查看（前端不用改，JSON 直出）。

## 验收

- 普通陈述句（「我今天去了趟医院」，不点名无问号）→ 至少 `min_responders` 条回复
  （**这是本单的核心回归测试，此前缺位**）。
- backchannel（「嗯」「哦哦」）且 `allow_silent_rounds=True` → 可静默；
  `allow_silent_rounds=False` → 仍保底回复。
- vocative 点名 → 永不静默（既有行为）。
- 85 全套测试（分档/引子/valence）零回归；`pytest -n auto`。
- 文档：`docs/stage.md` 运行合同的静默轮描述补一句语义（「仅 backchannel 可静默」）。
