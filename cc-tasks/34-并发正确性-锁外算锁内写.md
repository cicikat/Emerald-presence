# Brief 34 · 并发正确性:锁外算锁内写 + executor + 两处副作用修复

> 来源:审计 §1.1 / §1.2 / §五(run_test.py)/ §四(发送后写记忆),裁定见
> docs/critique-triage-20260708.md。四项都是"不改功能、只还正确性"的净收益。

## 1. detect_emotion 挪出 uid_lock(审计 §1.2,已核实)

`core/pipeline.py::post_process`(~905 起):`detect_emotion`(8s 超时的网络调用)
当前在 `uid_lock(user_id)` 临界区内。改为:

1. 进锁**前** `await detect_emotion(reply)`(保留 8s 超时与 fail-open→"neutral")。
2. 锁内只做既有的读-改-写,使用锁外算好的 `_emotion`。
3. 语义确认:emotion 只依赖 `reply` 文本,不依赖锁内读到的任何状态(cc 核对
   detect_emotion 入参确无锁内数据依赖后再动)。
4. 锁内其余部分(profile 条件判断等纯本地逻辑)不动。

## 2. vector_store 阻塞段 executor 化(审计 §1.1,方向已核实)

`core/memory/vector_store.py` 的 `query` / `upsert`(以及同文件其他 sqlite 内联段):

- 同步 sqlite 段包 `asyncio.to_thread(...)`;调用点 `core/pipeline.py::fetch_context`
  相应改 await(cc 全仓 grep 调用点,逐个接)。
- **sqlite 线程安全**:确认连接是否 per-call 新建;若存在长连接复用,改为线程内
  自建连接或 `check_same_thread=False` + 串行 executor(单 worker `ThreadPoolExecutor`
  更稳,拍板用单 worker——向量库本就低频,串行足够,免去并发写锁问题)。
- 同步入口(若有非 async 调用方)保留原函数,新增 async 包装,不破坏现有签名。

## 3. run_test.py 停止改写 config.yaml(审计 §五)

现状:运行时写入再删除 `data_prefix` 行,中断即留下指向 test_sandbox 的生产配置。
改为环境变量:`run_test.py` 设置 `YEXUAN_DATA_PREFIX=data_test_sandbox`(命名 cc 对照
sandbox/config_loader 现有约定),`core/sandbox` / `config_loader` 读取顺序:env > config。
config.yaml 从此只读。删除原写入-回滚代码及其注释里的 P0-1 疤痕说明,known-issues 对应条目关闭。
