# X1 · 向量库决策备忘

> 决策文档，不是施工单。目的：把「要不要建向量库 / 建哪种」一次定清，给 P3（画像召回）、X2（query relevance 评分）、G3（溯源）、X3（自己上网）解锁实现方向。
> 读者是代码小白，名词第一次出现会用一句话解释。**结论在最后第 6 节**，急的话直接翻到底。
> 你已倾向 sqlite-vec、约束是「包体敏感 + 聊天数据敏感 + 未来后端上服务器」。本文在此基础上做了核对，有一处需要纠正你（和 sonnet）的一个前提。

---

## 1. 先纠一个前提（重要）

你转述 sonnet 说「sqlite-vec 应该能和现有的结合上」。我核对了代码，**现状不是这样，但结论反而更简单**：

- 现在所有记忆**没有用任何数据库**，全是**散文件**：`episodic.json`、`memory_index.json`、`event_log/*.md`（共 90 个 .md），`data/` 总共 47MB。
- 也就是说**没有一个现成的 sqlite 让向量库去"接上"**。

所以正确的画面是：**向量库是一个全新的、独立的"索引文件"，挂在现有散文件旁边**。这其实是好事，而且是我推荐的架构（见第 4 节）：

> JSON / Markdown 继续当**真相源**（source of truth）；向量库只是一个**可随时删掉重建的检索索引**。索引坏了不影响你的真实记忆数据。

这条很关键，因为它意味着上向量库**风险很低**——它不碰你的真实数据，只是旁边加一个加速检索的东西。

---

## 2. 真正要决策的，不是 sqlite-vec vs chroma

向量检索分两半，很多人（包括方案对比）只盯着后一半：

| 半 | 干什么 | 你纠结的 |
|---|---|---|
| **① embedding 来源** | 把一句中文文本 → 一串数字（向量）。**这一步需要一个模型** | ← 没人提，但这才是大头 |
| **② 向量存储/检索** | 把向量存起来，查询时算"谁离我最近"（语义相似） | ← sqlite-vec / chroma 之争 |

**核对结果：现在整个后端没有任何 embedding 能力**——`requirements.txt` 里没有 torch、没有 sentence-transformers、没有本地模型，什么都没有。

所以"建向量库"真正的成本和取舍，**80% 在 ① embedding 来源**，因为它直接撞上你两条约束：

- **包体敏感**：本地跑一个中文 embedding 模型（如 bge-small-zh）+ 运行时 ≈ **几十到一百多 MB**，这会把包撑大。
- **聊天数据敏感**：用别人的 embedding API（云端）= **把聊天文本发给第三方**，隐私上不舒服。

这两条**互相打架**：怕包大 → 想用 API；怕数据外泄 → 想本地。这才是 X1 的真问题。第 5 节给分阶段解法。

---

## 3. 存储层：sqlite-vec vs chroma（同意你/sonnet，选 sqlite-vec）

先把②定了，它简单。

| 维度 | **sqlite-vec**（推荐） | chroma |
|---|---|---|
| 形态 | 一个 `.db` 文件，嵌进进程，**无后台服务** | 要么嵌入式 persistent client，要么跑一个**独立服务进程** |
| 依赖体积 | 极小（一个 sqlite 扩展） | 较重（拉一堆传递依赖） |
| 和你现有架构 | 完美——你本来就是"一堆文件"，再多一个 `.db` 旁挂，零心智负担 | 多一层"集合/服务"概念，和你的散文件风格不搭 |
| 你数据规模下的性能 | 47MB、估计低至几千条向量，**暴力全扫**就够快，毫秒级 | 它的 ANN 近似索引在你这个量级**用不上**，是为百万级设计的 |
| 上服务器后 | 照样是个文件，随后端一起走，零运维 | 多一个要部署/守护的服务 |

**关于你纠结的"chroma 有分区和时间管理，需不需要"**：不需要，原因是你已经有了——

- chroma 的"**分区**"（collection / 按 namespace 隔离）→ 你天然按 `char_id` + `uid` 分用户/分角色。在 sqlite-vec 里这就是表里两个**普通列**，查询时 `WHERE char_id=? AND uid=?` 一过滤就完成隔离，比 chroma 的集合更灵活。
- chroma 的"**时间管理**"→ 你每条记忆本来就带 timestamp。在 sqlite-vec 里它是一个 `ts` 列，"只召最近 / 时间衰减加权"就是一句 `ORDER BY` 或在打分里乘个衰减系数。**这正好是 P3（画像只召最近）和 X2（评分）要的**。

> 一句话：chroma 的卖点你**用普通的列 + WHERE/ORDER BY 就实现了**，还更可控。在你的规模和"文件即一切"的风格下，**sqlite-vec 是更省事的选择**。✅

## 4. 推荐架构：向量库当"可重建索引"，不当真相源

```
真相源（不变）           索引层（新增，可重建）         查询路径
─────────────           ──────────────────           ──────────
episodic.json     ──┐                                用户消息
event_log/*.md    ──┼──► embed() ──► sqlite-vec ◄──── → embed(query)
profile 偏好条目   ──┘    (①来源待定)   .db 索引          → KNN + 元数据过滤(char_id/uid/ts)
                                                       → 候选
                                                       → 与现有 强度×衰减 融合打分(X2)
```

要点：

1. **JSON/MD 永远是真相**；sqlite-vec 只存 `(id, 向量, char_id, uid, ts, 文本指针)`。**索引坏了/换模型了 → 删库重建**，不丢任何记忆。这对小白最友好：向量库出事不会毁数据。
2. **tag 不废，降级为"粗筛"**：现在 tag 粒度粗（你原话），改成 tag 先粗筛候选集，向量在候选里做**细排**。tag 和向量是互补不是替代。
3. **embedding 藏在一个边界后面**：全后端只通过一个 `embed(texts) -> vectors` 的小接口拿向量。这样**①来源可随时换**（本地↔API↔自托管）而上层代码不动——这是应对"现在本地、将来上服务器"最省力的设计。
4. **一个回填任务**：上线时跑一次，把已有 episodic + event_log 批量 embed 进库；之后每次写记忆顺手 embed 一条。

---

## 5. embedding 来源：分阶段（解你"包体 vs 隐私"的死结）

把第 3 节的 `embed()` 接口做出来后，来源就是可插拔的。建议分两阶段：

| 阶段 | 环境 | 推荐来源 | 包体 | 隐私 |
|---|---|---|---|---|
| **A（现在，本地开发）** | 你本机 | **二选一，见下** | — | — |
| **B（未来，后端上服务器）** | 你的服务器 | **自托管 embedding 端点**（模型跑在你自己服务器，后端用 HTTP 调） | 客户端零负担 | 数据不出你的机器 ✅✅ |

**阶段 B 是终态最优解**：模型部署在你自己服务器上，既不撑客户端包体，聊天数据也不出你的基础设施。因为有第 4 节的 `embed()` 边界，从 A 切到 B **只是改一行配置**。

**阶段 A 现在二选一**（这是唯一需要你拍的点，因为牵涉你对"开发期自己的聊天数据"的隐私底线）：

- **A1：OpenAI 兼容 embedding API**（如硅基流动/各家的 `bge`/`text-embedding` 端点）。
  - 好处：**零新依赖**——你 `requirements.txt` 里已经有 `openai` SDK，直接复用，包体一点不涨。开发期最快。
  - 代价：开发期你自己的聊天文本会发去这个 API 商。
- **A2：本地 ONNX 小模型**（bge-small-zh-v1.5，onnx int8 ≈ 30–45MB + onnxruntime ≈ 15MB）。
  - 好处：**完全本地、零外发**，开发期数据也不出门。
  - 代价：包体 +约 50–100MB（撞你"包体敏感"，但这是开发期临时的，上线走阶段 B 后客户端不带它）。

> 我的默认建议：**阶段 A 选 A1（API）**。理由：① 包体零增长，正合你敏感点；② 开发期外发的只是**你自己的**测试数据、量小、且终态（阶段 B）本来就会换成自托管不外发；③ 真正的隐私终态由阶段 B 保证。除非你连开发期自己的聊天也完全不接受外发，那就 A2。

---

## 6. 结论（TL;DR）

1. **建，值得建**：当前召回是字符 n-gram 关键词重叠（`hit/len(keywords)`）算相关性，粒度粗是结构性的，向量库能根治，且**风险低**（它是可重建索引，不碰真相源）。
2. **存储层：sqlite-vec**。同意你和 sonnet。chroma 的分区/时间功能你用普通列 + `WHERE/ORDER BY` 就有了，在你的规模和"文件即一切"风格下 sqlite-vec 更省事。
3. **纠正一个前提**：没有现成 sqlite 可"接上"——向量库是**新增的旁挂索引文件**（反而是更干净的架构）。
4. **真正的大头是 embedding 来源**，不是存储。把它**藏在 `embed()` 接口后面**，做成可插拔。
5. **分阶段**：现在本地用 **API（A1，包体零增长）**；上服务器后切**自托管端点**（数据不出门）。切换只改配置。
6. **tag 不废**，降为粗筛；向量做细排；时间/recency 用 `ts` 列实现——直接喂给 P3 和 X2。

### 需要你点一个头

**只有一个**：阶段 A 的 embedding 来源——
- **A1 API**（推荐，包体零增长，开发期自己的测试数据会过第三方）
- **A2 本地 ONNX**（零外发，但开发期包体 +50–100MB）

点完这个，X1 就能展开成施工 brief，并解锁 P3 / X2 的实现方向（两者都依赖这里定的"语义相似度"路径）。

> 顺带：**X3（叶瑄自己上网、自建库、自检索）大概率复用同一个 sqlite-vec + `embed()`**——网页检索结果 embed 进同一套索引，自然就有了"自己的可检索资料库"。等 X3 决策备忘时再展开，但存储选型在这里已经替它定了。
