## 全局背景：为什么电脑端"看起来"快很多

实读后，电脑端和安卓端走的是**两套完全不同的下行通道**，这是所有"手机比电脑慢"现象的总根因：

| | 电脑端（Emerald-client） | 安卓端（yexuan_memery） |
|---|---|---|
| 下行方式 | **WebSocket 流式**（`message_stream_delta` 逐 token 推） | **HTTP 轮询** `/mobile/poll`（无流式） |
| 首字延迟 | 生成开始即逐 token 显示（~1-2s 出第一段） | 必须等**整条回复生成完**才拿得到 |
| 投递延迟 | WS 实时推，几乎为 0 | 前台 5s 定时轮询 / 后台依赖中继 SSE |

参考 `docs/channels.md` →「桌宠流式输出协议（Spec #9）」与「手机端轮询通道」。后端 `admin/routers/chat.py:101` 明确：`_use_stream = (channel_name == "desktop") and _dws.is_connected()`——**只有 desktop channel 且 WS 在线才流式，mobile / QQ 永远非流式**。

所以"电脑端响应比安卓快半分钟"其实是两件事叠加：
1. **安卓没有流式**：电脑端在生成过程中就已经在逐字显示了（"后端已经有输出"指的就是这个），而安卓要等 `run_llm` 整条返回。长回复时这一项就能差出几十秒。
2. **安卓投递有轮询周期**：整条生成完之后，安卓还要等下一个轮询周期 / 中继信号才拿到。

下面分三个问题给具体落点。

# 问题二：换行气泡未切割（触发器 + 电脑转发，同一根因）✅ 最小改动先做

## 现象
- 触发器（调度器主动消息）同步到手机时，多行内容挤成**一个气泡**，没按换行切开。
- 手机电脑同时在线、往电脑发消息，转发到手机的那条同样不切割。
- 但：**手机本端发消息**得到的回复、以及**历史记录**都是正常按换行切割的。

## 根因（实读 `D:\ai\yexuan_memery`）

安卓其实**已经有**换行切割函数 `_splitReplySegments`（`lib/pages/app_shell.dart:1282`，按 `RegExp(r'\r?\n+')` 切）。问题是**只有部分路径用了它**：

| 路径 | 函数 | 是否切割 |
|---|---|---|
| 手机本端发消息的回复 | `_appendHimReplySegments`（`app_shell.dart:1260`）→ 调 `_splitReplySegments` | ✅ 切 |
| 历史记录加载 | `_splitHistorySegments`（`app_shell.dart:1368`）→ 调 `_splitReplySegments` | ✅ 切 |
| **轮询收到的主动消息**（触发器 + 电脑转发） | `_pollMobile`（`app_shell.dart:1154-1156`）→ **`message.toChatMessage()` 直接整条塞进一个气泡** | ❌ **不切** |

关键代码 `lib/pages/app_shell.dart:1154-1156`：
```dart
final incoming = freshMessages
    .map((message) => message.toChatMessage())   // ← 一条 poll 消息 = 一个气泡
    .toList(growable: false);
```
而 `MobilePollMessage.toChatMessage()`（`lib/models/app_models.dart:837`）只是把整个 `content` 原样塞进单个 `ChatMessage`，完全没切。

> 注意：触发器消息和"电脑转发"消息走的是**同一条** `/mobile/poll` 通道（后端 `record_assistant_turn(fanout="all")` 把电脑这一轮的回复也写进了 mobile 队列，见 `core/turn_sink.py:_fanout`），所以茶茶说"这俩是同一个问题"完全正确——修一处即可。

## 修复（纯安卓前端，`lib/pages/app_shell.dart`）

把 `_pollMobile` 里 `incoming` 的构造从"每条消息一个气泡"改成"每条消息按换行拆成多个气泡"。建议复用已有的 `_splitReplySegments`，并保留时间戳 / behavior 语义。

### 实现要点
1. 把 `freshMessages.map(toChatMessage)` 改为 `expand`，对每条消息的 `content` 调 `_splitReplySegments`，每段生成一个 `ChatMessage`（role `'him'`、time 用消息自身时间戳）。
2. **去重 / ack / seenIds 不变**——它们以整条 poll 消息（`msg.id` / `seq`）为单位，切割只发生在"显示"层，不影响对账。`_isRecentAssistantReply` / `_rememberAssistantReply` 仍按整条 `content` 处理（保持现状即可，循环里不要逐段去重）。
3. 行为消息（带 `behavior`，如锁屏/外卖 overlay）通常是单行，切割对它们无副作用；如担心，可仅当 `behaviorKind` 为空时才切割、否则保持单气泡。

### 参考改法（示意，CC 按实际类型签名调整）
```dart
// app_shell.dart，替换 1154-1156 的 incoming 构造
final incoming = <ChatMessage>[];
for (final message in freshMessages) {
  final base = message.toChatMessage();
  final parts = _splitReplySegments(message.content);
  if (parts.length <= 1) {
    incoming.add(base);
  } else {
    for (final part in parts) {
      incoming.add(ChatMessage(role: base.role, text: part, time: base.time));
    }
  }
}
```
> 进阶（可选）：要做到和本端发消息一样"逐段带打字间隔地冒出来"，可以在 poll 路径里改用类似 `_appendHimReplySegments` 的逐段 `setState` + 随机延迟。但首版只要拆成多气泡、一次性 `addAll` 即可解决茶茶的诉求，逐段动画属锦上添花。

### 验证
- 后端用 `POST /mobile/push`（带换行 `content`，如 `"第一句\n第二句\n第三句"`）推一条，手机在线时应显示成 **3 个气泡**。
- 触发器主动消息、电脑端发消息转发到手机，均应按换行切割。
- 跑 `flutter test`（先设 `NO_PROXY`），重点确认 `test/foreground_mobile_delivery_contract_test.dart` 仍过；如该契约测试断言了"一条 poll = 一条消息"，需同步更新断言为按换行展开。

# 问题一：手机端比电脑端慢 ~30s

这个问题有**两个场景**，根因不同，建议都处理。

## 场景 A：从手机发消息，手机自己显示得慢

### 根因
安卓 `BackendClient.sendChat`（`lib/services/backend_client.dart`）打的是 **`/desktop/chat`**，且是一个**阻塞的 HTTP 请求**（timeout 120s），**等整条回复生成完才返回**。与此同时，如果电脑端 WS 在线，后端对这同一轮走流式（`chat.py:101-112`）逐 token 推给电脑——所以电脑端 1-2s 就开始显示，手机端要干等整条生成完（外加问题三的探针延迟）。这就是"半分钟"的主要来源。

### 修复方向（按投入从小到大）
1. **立刻能做、低风险**：发送后**先本地乐观显示用户气泡 + "对方正在输入"指示**（`_himTyping`），让等待有反馈；这不缩短真实延迟但显著改善体感。检查 `_sendToBackend`（`app_shell.dart:1211`）是否在 await 之前就已 setState 出 typing 态。
2. **中等投入、治本**：给安卓加**流式下行**。后端已有流式能力，但目前只对 `channel=="desktop"` 且走 WS 的连接开放。两条路线：
   - (a) 让安卓也接 `/ws/desktop`（带 `Authorization: Bearer`，见 `docs/channels.md` 鉴权段），复用 `message_stream_*` 帧。代价：要在 Flutter 实现 WS 客户端 + 流式气泡。
   - (b) 后端给 mobile 开一个 **SSE 流式回复端点**（如 `/mobile/chat/stream`），把 `run_llm_stream` 的增量按 SSE 吐给手机。代价：后端加端点 + 安卓加 SSE 读取。
   - 评估：(a) 与电脑端协议一致、改后端最少，推荐优先评估 (a)。
3. **附带**：安卓 `sendChat` 用 `/desktop/chat` 在语义上也怪（手机消息以 desktop channel 入 pipeline）。是否应改用 `/mobile/chat` 需产品确认——但注意 `/mobile/chat` **也不流式**，单改它不解决慢的问题，只是语义更正。

## 场景 B：从电脑发消息，转发到手机慢（手机被动收）

### 根因（实读后端 + 安卓原生层）
后端侧**不慢**：`record_assistant_turn(fanout="all")` 在 `chat.py:139` 是**在 conversation_lock 内、生成后立即 await** 的，mobile 队列写入 + 中继信号（`channels/relay_publisher.schedule_signal_publish`）几乎同步发生。慢在**手机取件节奏**：

**前台（App 在前台）**：
- `MainActivity.onResume`（`MobileNotificationService` 同包 `MainActivity.kt:45-53`）会停掉原生后台服务、置 `mobileAppInForeground=true`。
- 于是 Flutter 的 5s 定时器 `_startMobilePollTimer`（`app_shell.dart:199-201`）生效，调 `_pollMobileIfRelayUnavailable` → `_pollMobile`。
- **问题**：`BackendClient.pollMobile`（`backend_client.dart`）**从不传 `wait` 参数**，即 `wait=0`，是**短轮询**。所以一条消息最坏要等满一个 5s 周期才被取到。
- 后端其实**支持长轮询**：`/mobile/poll?wait=55`（`channels/mobile.py:poll` 用 `asyncio.Condition`，入队即唤醒，见 `_write_to_queue` 的 `notify_all()`）。一旦用长轮询，消息入队**毫秒级**就返回。

**后台（App 不在前台 / 息屏）**：
- 原生 `MobileNotificationService` 接管，依赖中继 SSE（`connectAndReadRelay`，`MobileNotificationService.kt:214`）。
- **若中继已配置且连上**：SSE 信号 → `triggerRelaySignalPoll` → 立即拉取，基本实时。
- **若中继未配置**：`startConsumerCoordinator`（:151）走 `runOneShotPoll` 后，补偿轮询间隔是 `supplementalPollIntervalMs = 6 * 60 * 60 * 1000`（**6 小时**！`MobileNotificationService.kt:45`）——后台基本收不到，直到下次打开 App。
- 中继断线重连用指数退避，最高 `maximumRetryDelayMs = 60_000`（60s），断连窗口内来的消息要等下次重连/补偿，**这正好能解释 ~30s 量级的抖动**。

### 修复

**B-1（强烈推荐，治前台延迟，改动小）**：让 Flutter 前台轮询改用**长轮询**。
- `backend_client.dart` 的 `pollMobile` 增加 `int waitSeconds = 0` 参数，拼到 query：`/mobile/poll?limit=$limit$afterQuery&wait=$waitSeconds`；同时把该请求的 HTTP `timeout` 调到 > wait（如 `wait=25` 配 `timeout=35s`）。
- `app_shell.dart` 前台调用处传 `waitSeconds: 25`（或 55）。可保留 5s 定时器作兜底，但长轮询在位时实际是"挂起等待、入队即返回、立刻再发起下一次"，延迟从 ≤5s 降到 ~0。
- 注意：`_pollingMobile` 重入锁（`app_shell.dart:1113`）已存在，长轮询期间不会叠发，正确。

**B-2（治后台延迟）**：
- 把 `supplementalPollIntervalMs` 从 6 小时调到一个合理值（如 5-15 分钟），作为**中继未配置/断线时**的兜底，避免后台几小时收不到。（`MobileNotificationService.kt:45`）
- 确认中继三件套已配置：后端 `config.yaml` 的 `relay_base_url` / `relay_topic` / `relay_token`（见 `docs/channels.md`），以及安卓侧 `setRelayBaseUrl/Topic/Token`（`app_settings_store.dart:611+`）。**中继是后台实时性的关键**，未配置时后台必然慢。
- 可选：缩短中继重连最大退避（`maximumRetryDelayMs` 60s → 如 15s），降低断连窗口内的最坏延迟。

### 验证
- 前台：电脑发一条，手机在前台应在 ~1s 内出现（长轮询生效后）。对比改前的最坏 5s。
- 后台：配好中继后息屏发一条，应几秒内出通知；断网模拟中继不可用时，确认 B-2 的兜底间隔内能补到。
- 跑 `flutter test`，关注 `test/foreground_mobile_delivery_contract_test.dart` / `test/android_relay_signal_contract_test.dart`。

# 问题三：后端输出提速（专项）

目标：缩短"用户发出 → 第一个 token / 整条回复"的后端耗时。实读 pipeline 后，**最大的一处可省延迟是 owner-chat 路径上的工具探针**。

## 找到的瓶颈

### 瓶颈 1（重点）：owner-chat 探针是串行全模型 LLM 往返，且未路由到轻量 preset
`admin/routers/chat.py` 的 owner-chat 流程在 conversation_lock 内**先**跑 `_probe_and_execute_tools`（`chat.py:78`），**再** `fetch_context` → `build_prompt` → 生成。探针本体 `chat.py:225`：
```python
probe_raw = await _llm.chat(probe_messages, tools=tools_schema)   # ← 没有 call_category="probe"
```
两个问题：
1. **未传 `call_category="probe"`**：`core/llm_client.py` 里 `_CALL_TIMEOUTS["probe"]=10.0`，且 probe 类别在 `model-presets` 里路由到**轻量模型**（见 `docs/model-presets.md`）。这里默认成了 `"chat"` 类别 → **探针用的是重对话模型 + 默认更长超时**。对照 QQ 路径 `main.py:443` 是**有**传 `call_category="probe"` 的——owner-chat 这里是漏配。
2. **没有关键词快路径**：QQ 路径在跑 LLM 探针前有 `will_skip_probe` 的轻量关键词判断（`main.py:387` 附近），owner-chat 路径**每一轮都无条件打一次探针 LLM**，即使消息明显没有工具意图。

→ 结果：每条 owner（电脑/手机）消息，在真正生成前都白白串了一次"重模型"往返。这对电脑端拖慢首字、对手机端拖慢整条。

#### 修复
1. **立刻、低风险**：给 `chat.py:225` 的探针调用补上 `call_category="probe"`：
   ```python
   probe_raw = await _llm.chat(probe_messages, tools=tools_schema, call_category="probe")
   ```
   确认 `model-presets` 里 `probe` preset 指向轻量/快模型；若没有该 preset，按 `docs/model-presets.md` 补一个。
2. **中等收益**：把探针与 `fetch_context` **并行**。二者无依赖（探针只用 short_term + 当前消息，`fetch_context` 读各记忆层），探针结果 `tool_result_text` 只在 `build_prompt` 时才需要。用 `asyncio.gather` 同时发起，可把探针耗时藏进 `fetch_context` 的耗时里。
   - 注意：探针里若命中 side-effect 工具会 `execute`，需确认与 `fetch_context` 并发无共享状态写冲突（探针 execute 的 origin 是 `user_live`，多为 info/desktop 工具）。先评估再做。
3. **可选**：移植 QQ 路径的关键词快路径到 owner-chat，明显无工具意图时直接跳过探针 LLM。收益最大但要小心漏判工具触发，建议放在 1、2 之后。

### 瓶颈 2：手机端整条等待（已在问题一覆盖）
mobile 不流式（`chat.py:101`），所以"提速"对手机端的体感主要靠问题一的流式化。后端侧 `run_llm`（`core/pipeline.py:534`）带 `with_retry`，正常路径无额外开销；流式 `run_llm_stream`（:545）已就绪，缺的是 mobile 侧的接入（见问题一场景 A 修复 2）。

### 瓶颈 3（小项，确认即可）：prompt 组装与裁剪
`build_prompt` 的 20k 字符硬上限裁剪（`docs/prompt-layers.md` 的裁剪顺序 `event_search → mid_term → diary → episodic → lore`）是 CPU 操作，通常非主要耗时。**不建议**在本单动它，除非 profiling 显示 `build_prompt` 单独耗时显著。

## 验证（问题三）
- 在 `_probe_and_execute_tools` 前后加耗时日志，确认补 `call_category="probe"` 后探针耗时下降（应接近 10s 超时档内的轻量模型实际耗时，通常 1-2s）。
- 跑 `python tests/run_eval.py`（如改动触及 tag/probe 行为）与 `pytest`，确认工具触发率不回退。
- 端到端：同一条消息，记录"发送 → 首字（电脑流式）"和"发送 → 整条返回（手机 HTTP）"的耗时，对照改前。
