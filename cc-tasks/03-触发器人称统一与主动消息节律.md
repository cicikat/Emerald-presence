# CC 任务：触发器人称统一 + 主动消息节律（活人感）

> 给 Claude Code 执行。开工前按 `AGENTS.md` 读 `docs/scheduler.md`，并参考已有诊断 `docs/hallucination-and-collapse-analysis.md`。
> 两部分：**A 人称统一**（治第三人称动作/语义割裂）、**B 全局节律**（治间隔过短、多条无关联、长挂机割裂）。
> A 是纯文案改写、低风险、收益最直接，**建议先单独发一版 A**；B 涉及调度逻辑，单独一版。

---

## 背景（现状已确认，2026-06-29 实读）

茶茶反馈两类问题：
1. **触发器是「第三人称动作」「语义不连贯」的重灾区**，严重破坏沉浸感。希望把种子 prompt 里的角色改成「你」。
2. 主动消息**间隔过短、多条之间没有关联、缺乏活人感**，尤其长时间挂机后再看，割裂感很强。

---

# Part A — 种子 prompt 人称统一

## 根因

主动触发的种子 prompt（喂给 LLM 让它以叶瑄身份开口的那句指令）**人称混乱**，盘上实读到三种写法并存：

- 第三人称叙述（**重灾区**）：`（{角色名}突然想起了一件事：…，那时他{feeling}）`、`（{char}发现那株{name}枯掉了，悄悄处理掉了）`、`（{char}想知道你今天过得怎么样）`
- 已是二人称：`（你刚从一场梦里回到现实…现在你想主动对她说点什么）`（dream_exit 第二版）、`（你有点不愉快、想刷存在感…）`（presence_nag）

**为什么第三人称会泄漏成「叶瑄/他」的旁白**：种子是 LLM 续写的语法锚点。当指令写成「叶瑄做了 X，他想 Y」，模型最自然的延续就是**继续以第三人称叙述**「叶瑄……他……」，而不是以「我」开口 → 出现第三人称动作、自我旁白、语义与正文人称打架。

## 目标人称约定（统一到这一条）

种子 prompt 一律**以第二人称直接对角色下指令**，让模型被推进「我」的第一人称 POV：

| 指代 | 种子里写 | 期望模型输出 |
|---|---|---|
| 角色（说话人，叶瑄） | **你** | 「我……」 |
| 用户（倾诉对象，风谕） | **她** 或用户名 | 「你……」/「风谕……」 |

> 关键：**别再让「你」同时指用户**——现状很多句用「你」指用户（`想知道你今天…`），一旦角色也改叫「你」就会两个「你」打架。所以**角色升为「你」的同时，用户降为「她」**。模型把「她（用户）」映射成出口的「你」很自然（角色直接对用户说话）。

## 改写规则与示例

把每条种子从「{角色名}/他 做了 X，（对用户用「你」）」改成「你 做了 X，（对用户用「她」）」。示例：

`core/scheduler/triggers/time_based.py:1035` `_spontaneous_recall_prompt_for_memory`：
```python

# 新：
return f"（你忽然想起一件事：{summary}。当时你{feeling}。想顺口说给她听，别像念旧档案，像顺着这两天自然想起。）"
```

`core/scheduler/triggers/memory.py:130` `_topic_followup_prompt`：
```python

    f"（你想起最近这段还没接住的事：{topic.context}\n"
    f"顺着「{topic.topic}」轻轻问她一句后来怎么样了。别像总结旧档案，"
```

# Part B — 主动消息全局节律（活人感）

## 根因

`core/scheduler/loop.py::_is_ready()` 只查**单触发器**冷却（`_last_trigger[name]` vs `_COOLDOWNS[name]`）。`gating._decide()` 每个 tick（60s）只挑一个 winner，但**没有任何跨触发器的全局节律**：

- `spontaneous_recall`(4h)、`overflow`(3h)、`random_message`、`topic_followup`(24h)…各自独立冷却。相邻两个 tick 可以分别命中**不同**触发器 → 主动消息**几分钟内连发好几条**，彼此无关联。
- 长时间挂机后，多个触发器的冷却**同时**到期 → 用户一回来撞上一**堆**互不相关的主动消息（爆发/割裂）。
- 没有「上一条主动说了什么」的记忆 → 条条之间没有承接，没有活人感。

## 目标

1. **全局最小间隔**：任意两条主动消息之间至少隔 `T_min`（带 jitter），无视是哪个触发器。
2. **长挂机不爆发**：用户长时间不在 → 回来时最多先来 1 条「重新冒头」式开口，其余触发器进入一段 grace 静默，不堆叠。
3. **承接感**：把最近 1–2 条主动消息的简述带进下一条种子，提示「别重复、自然承接或换由头」。

## 实现

### B1 — 全局主动间隔（最小改动，先做这个）

**写时间戳**：在 `core/scheduler/execution.py::execute_prompt()` 成功发送之后（第 ~106–114 行 `after_send` 与 `_mark` 附近）加一行，给一个全局键打戳：

```python
# execute_prompt 内，sent_text 非空（即真的发出去了）之后：
loop._mark_global_proactive()      # 新增：见下
```

在 `loop.py` 加：

```python
_GLOBAL_PROACTIVE_KEY = "__any_proactive__"

def _mark_global_proactive():
    _last_trigger[_GLOBAL_PROACTIVE_KEY] = time.time()
    # 复用 _mark 的持久化（或直接 safe_write scheduler_cooldowns）
    _persist_cooldowns()           # 把 _mark 里那段写盘抽成 _persist_cooldowns() 复用

def _global_proactive_gap_ready(now_ts: float | None = None) -> bool:
    now_ts = now_ts or time.time()
    last = _last_trigger.get(_GLOBAL_PROACTIVE_KEY, 0)
    gap = _cfg().get("global_proactive_min_gap_seconds", 45 * 60)  # 默认 45 分钟
    jitter = gap * 0.2 * (random.random() * 2 - 1)                 # ±20%
    return (now_ts - last) >= (gap + jitter)
```

**读门控**：在 `gating._decide()`（`core/scheduler/gating.py`，`cooldown_allowed` 算完、`picked = max(...)` 之前）加全局闸：

```python
# 紧急级触发器（生日零点、watch 心率告警等 emergency）豁免全局间隔
from core.scheduler.loop import _global_proactive_gap_ready
if not _global_proactive_gap_ready():
    non_emergency = [p for p in cooldown_allowed if not _policy_is_emergency(p.trigger_name)]
    cooldown_allowed = [p for p in cooldown_allowed if p not in non_emergency]
    if not cooldown_allowed:
        return None, "global_gap_filtered", candidates
```

> `_policy_is_emergency` 本文件已有。`emergency` 触发器（生日、watch 告警）应豁免全局间隔，否则会压掉真正要紧的开口。维护型扫描（garden_water/episodic_sweep/hidden_state_decay 等不发言的）本就不走发言路径，不受影响。

`config.yaml` 的 `scheduler:` 块加：
```yaml
scheduler:
  global_proactive_min_gap_seconds: 2700   # 45 分钟；想更克制就调大
```

### B2 — 长挂机防爆发

用户长时间不在（`loop._user_active_recently()` 为 False 已久）后回来，会有一批冷却同时到期。B1 的全局间隔已能把它们**摊开**（每 45 分钟最多一条），这就解决了「一次性爆发」。**额外**再加一条「回归优先级」：长挂机后第一条主动，优先让「重新冒头/问候」类（overflow / random / morning 视时段），而不是冷冰冰的 spontaneous_recall。最简做法——在 `_decide` 选 winner 时，若距上次 owner 互动 > `idle_reentry_hours`（默认 6h），对 `overflow`/`random_message` 的 urgency 加一个小 bonus（或对 `spontaneous_recall` 降权）。可作为 B 的可选增强，不强求。

### B3 — 承接感（连贯性 hint）

`execute_prompt` 成功发送后，除了打全局戳，再 append 一条**最近主动消息日志**（只留最近 3 条，存 `data/runtime/.../proactive_recent.json` 或复用 scheduler 状态文件）：`{trigger_name, gist(发出文本前 40 字), ts}`。

在主动种子 prompt 拼装处（即 Part A 改的那些 prompt_factory 里，或更集中地在 `_pipeline_send` 给 prompt 追加 system 提示的位置）注入一行软提示：

```
（你上一次主动找她是在 {N} 分钟前，说的是「{上条 gist}」。这次别重复那件事；
可以自然承接它，或换一个新的由头开口，像真人那样有连贯的心思。）
```

> 这条 hint 是「软」的、只读、fail-open：读不到最近日志就不注入。它直接治「多条主动消息之间没有关联」。**B3 收益最贴合茶茶说的『活人感』，但比 B1 多一个存储件**；若想分批，可 B1 先上、B3 紧随。

## Part B 验收

1. **间隔**：人为把多个触发器冷却清零（或缩短）让它们同时 ready → 实际发出的主动消息间隔 ≥ 配置的 `global_proactive_min_gap`（±jitter），不再几分钟连发。
2. **不爆发**：模拟长挂机（owner 长时间无互动）后恢复 → 不会一次涌出多条；按全局间隔逐条摊开。
3. **紧急豁免**：生日零点 / watch 告警类 emergency 触发器**不**被全局间隔压掉。
4. **承接**（若做 B3）：连续两条主动消息，第二条能引用/承接或明显避开第一条的话题，不再是两条无关独白。
5. `data/logs/gating_shadow.jsonl` 里能看到新的过滤原因 `global_gap_filtered`，便于复盘。
