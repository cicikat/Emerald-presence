## 背景（现状已确认，2026-06-29 实读）

茶茶反馈：主记忆连贯，但**主动触发的消息只到电脑前端，手机前端几乎收不到**，触发器表现割裂。根因不在主动链路，而在 fanout 的「活跃通道」过滤：

### 链路与根因

1. 调度器主动发言走 `core/scheduler/loop.py::_pipeline_send()` → `record_assistant_turn(fanout="all", source=TurnSource.TRIGGER/SENSOR/WATCH)`。
2. `record_assistant_turn` 调 `core/turn_sink.py::_fanout()`（第 84 行）。当 `fanout in ("all","broadcast")` 时：
   ```python
   targets = registry.get_active()      # ← 只取「活跃」通道
   ```
3. `channels/registry.py::get_active()` 过滤 `c.is_active`。
4. `channels/mobile.py::MobileChannel.is_active`（第 39–43 行）：
   ```python
   if not self._active: return False
   return time.time() - self._last_seen <= _ACTIVE_TTL_SECONDS   # 120 秒
   ```
   `_last_seen` 只在手机调 `/mobile/poll`（`poll()` 里 `self.touch()`）时刷新。

**结论**：手机端只要超过 **120 秒**没来轮询（锁屏、切后台、网络抖动都算），`is_active` 即为 False → `get_active()` 不含 mobile → `_fanout` 根本不调 `mobile.send()` → 这条主动消息**永不写进 `mobile_queue.json`**。等手机下次 `/poll` 时，队列里压根没有这条——**永久丢失，无补偿**。

### 为什么电脑端不割裂、手机端割裂

桌宠走 WebSocket + **wake ledger（Path A delivery ledger，见 `docs/scheduler.md`）** 有断线重连补播；**手机端对 scheduler turn 没有任何补播机制**。所以同一条主动消息，电脑收得到、手机收不到 → 主记忆连贯但触发器表现割裂。

### 前端已核实：客户端补齐机制完整，问题 100% 在后端（已读 `D:\ai\yexuan_memery`，Flutter 薄客户端）

实读手机端（`lib/pages/app_shell.dart` + `lib/services/backend_client.dart` + `docs/protocols/mobile-channel.md`），确认**客户端该有的补齐能力全都有，无需改动手机端**：

- **持久化游标**：`_pollMobile()` 用 `settingsStore.loadLastAckedMobileSeq()` 取持久化的 `lastAckedSeq`，以 `GET /mobile/poll?after=<lastAckedSeq>` 拉取。**跨重启、跨离线都能按游标补齐**——只要消息还在服务端队列里。
- **三条投递路径，全部依赖服务端队列**：
  1. 前台每 5 秒 poll（`_startMobilePollTimer`，仅 `AppLifecycleState.resumed` 时跑；`paused`/`hidden` 即 `_stopMobilePollTimer` 停 → 这正是 120s 后 `is_active` 转 False 的来源）。
  2. 后台 ntfy SSE 中继推送为主路径；**中继 signal 不带正文，收到后立刻按 `lastAckedSeq` 调 `/mobile/poll` 拉正文**。
  3. 中继断连 15 分钟 → AlarmManager 补偿 poll，之后每 6h 续约。
- **去重**：按 `message.id`（= 后端 canonical `turn_id`）+ `seenMobileMessageIds`（持久化、capped 200）对账，**多路径不会重复渲染**。
- **relay signal 的源头**：`schedule_signal_publish(item)` 是在 `mobile.py::_write_to_queue` **内部**调用的。

> **致命推论**：消息没入队 → `_write_to_queue` 没跑 → **relay signal 根本不会发** → 后台 app 永远收不到推送，连「去 poll」的触发都没有；而前台/补偿 poll 即便跑了，`after=lastAckedSeq` 拉到的队列里也没这条。**三条路径同时哑火，全卡在「没入队」这一个点上。** 所以本单只改后端「让主动 turn 无条件入队」，客户端那套完整的拉取/补偿/去重机器就立刻全部生效。

### 关键观察：mobile 队列本来就是「durable + 拉取」设计，120s 活跃门用错了地方

- `mobile.py` 的队列 `_write_to_queue` 是**持久化**的：写盘 `mobile_queue.json`、带 `seq` 游标、`_prune_queue` 自动按 24h / 500 条上限裁剪、并 `schedule_signal_publish(item)` 给 relay 推送信号唤醒离线 app。
- 也就是说，「手机此刻是否在长轮询」(`is_active`) 只该决定**长轮询的即时响应**，**不该决定「要不要为手机留下这条消息」**。把消息留进 durable 队列、等手机回来拉，正是这套队列存在的意义。
- 现在的 bug 本质：`is_active`（即时投递门）被错误地复用成了 durable 入队的闸门。

## 目标

主动/系统侧的 assistant turn（`source != USER_CHAT`）即使手机离线，也**无条件写进 mobile 的 durable 队列**（并触发 relay 信号）。手机下次 `/poll` 按游标补齐，不再丢消息。

## 实现：`core/turn_sink.py`

### 步骤 1 — 把 `source` 透传进 `_fanout`

`_fanout` 当前签名（第 84 行起）没有 `source`。在 `record_assistant_turn` 里调用 `_fanout` 的地方（约第 251 行）已经能拿到 `source`，把它传进去：

```python
# record_assistant_turn 内，约 251 行
targets, failures = await _fanout(
    assistant_text=assistant_text,
    uid=uid,
    fanout=fanout,
    behavior=behavior,
    exclude_origin_channel=exclude_origin_channel,
    ws_msg_id=_ws_msg_id,
    char_id=char_id,
    source=source,          # ← 新增
)
```

`_fanout` 签名加 `source: "TurnSource | None" = None`。

### 步骤 2 — durable 手机兜底（核心）

在 `_fanout` 里，`fanout in ("all","broadcast")` 分支算完 `targets = registry.get_active()` 之后，补一段：**若本条是主动来源（`source != USER_CHAT`）、mobile 通道已注册、但不在 active targets 里，则把 mobile 追加为投递目标**（它的 `send()` 只是写 durable 队列 + 发 relay 信号，对离线手机安全）。

```python
from core.turn_sink import TurnSource   # 顶部已可引用，置于模块顶即可

if fanout in ("all", "broadcast"):
    targets = registry.get_active()
    if exclude_origin_channel:
        targets = [ch for ch in targets if ch.name != exclude_origin_channel]

    # ── durable 手机兜底（窄版：仅主动来源）：离线手机也要把消息留进队列，等它回来拉 ──
    # mobile.send() 写的是 durable、按 24h/500 条裁剪的队列 + relay 推送，对离线手机安全。
    # is_active 只表示「此刻是否在长轮询」，不该决定是否为手机留消息。
    # 仅对机器主动来源（TRIGGER/SENSOR/WATCH）兜底，避免给 owner 桌面对话发多余手机推送。
    _is_proactive = source is not None and source != TurnSource.USER_CHAT
    if _is_proactive and exclude_origin_channel != "mobile":
        mobile_ch = registry.get("mobile")
        already = any(getattr(t, "name", "") == "mobile" for t in targets)
        if mobile_ch is not None and not already:
            targets.append(mobile_ch)
```

> **范围选择（给 CC 的决策说明 — 已结合前端 relay 行为定调，推荐「窄版」）**：
>
> **★ 推荐：窄版（仅主动来源入队）。** 在上面 `if` 里再加来源判断，只对 `source != USER_CHAT` 的 turn 给手机留底：
> ```python
> from core.turn_sink import TurnSource  # 顶部已可引用
> _is_proactive = source is not None and source != TurnSource.USER_CHAT
> if exclude_origin_channel != "mobile" and _is_proactive:
>     mobile_ch = registry.get("mobile")
>     already = any(getattr(t, "name", "") == "mobile" for t in targets)
>     if mobile_ch is not None and not already:
>         targets.append(mobile_ch)
> ```
> **为什么是窄版**：前端实读确认，每次 `_write_to_queue` 都会发 relay signal → 后台原生服务会据此弹通知（受静音时段 + 30min 冷却闸门，但仍会响）。若用「宽版」把 owner 在**桌面**正聊着的每条助手回复也入队，手机会为「你正在桌面进行的对话」推送通知——打扰。窄版只补「机器主动找你」的消息，正是茶茶报的那个 bug，且无多余推送。
>
> **宽版（所有 all/broadcast 都入队）**作为可选：能顺带做「跨设备连续性」（手机离线时桌面聊的内容回来也能在手机看到），但带上述通知副作用。**若以后要做跨设备同步，建议单独成单、配合通知闸门一起设计，别在本单作为副作用引入。** 本单先上窄版。

> `exclude_origin_channel == "mobile"` 的判断是为了不把「手机自己刚发的 owner turn」又回灌给手机。**已核实**：`admin/routers/mobile.py::mobile_chat` → `run_owner_chat_turn(message,"mobile")`（`admin/routers/chat.py:29`）→ `record_assistant_turn(exclude_origin_channel=channel_name=="mobile", source=USER_CHAT)`（chat.py:142,146）。所以这条判断成立，无需再改 origin 值。

### 步骤 3 — 不要让兜底目标污染 active 语义

`_fanout` 末尾的 `if not targets: logger.warning("no active fanout targets")` 不受影响。`sent_targets` 现在会含 "mobile"，符合预期（确实投递了——投进了 durable 队列）。`ws_msg_id` 在 `name in ("desktop","mobile")` 时作为 `msg_id` 透传，已有逻辑正确复用。

## 验收标准

1. **离线补齐**：让手机端**停止** `/poll`（或 kill app）≥ 2 分钟 → 用调度器触发一条主动消息（或手动跑一个 proposer）→ 等手机重新 `/poll` → **能在返回的 messages 里按 `seq` 游标看到这条**。修复前这条会丢。
2. **不重复**：手机端**正在**长轮询时触发主动消息 → 手机只收到**一条**（不因为「active targets 已含 mobile + 兜底又 append」而重复）。`already` 去重保证这一点，务必验证。
3. **origin 不回灌 + 不打扰**：从**手机端**发一句话产生 owner 回复 → 该回复**不**被兜底塞回手机队列（`exclude_origin_channel == "mobile"`）；从**桌面端**正常聊天的 owner 回复（`source == USER_CHAT`）→ **不**入手机队列、手机**不**收到多余 relay 推送（窄版 `_is_proactive` 保证）。
4. **relay 信号链路**：离线主动消息场景下，`_write_to_queue` 末尾 `schedule_signal_publish(item)` 被调用 → 后台 app 收到 ntfy signal → 据 `lastAckedSeq` 回拉到这条。若服务端 `relay_base_url/relay_topic/relay_token` 未配（`relay_publisher._relay_config()` 返回 None），signal 静默跳过——此时仍可靠前台 5s poll 与 6h 补偿 poll 补齐（只是后台即时性差）。**要后台即时推送，需确认这三项已配 + ntfy 可达。**
5. **桌宠不回归**：桌面端主动消息照常即时到达，wake ledger 行为不变。
