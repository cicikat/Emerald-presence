# Brief 26 · 角色日记生成与主动发言解耦

> 问题:角色的每日内心日记(`_generate_and_store_diary`)只在 `daily_journal` 主动消息
> **真的发出去**时才写(挂在 proposer 的 `after_send` 回调上)。主动消息要过 gating 五道闸,
> 任何一道没过,当天日记静默丢失 → 用户观察到"偶尔写而不是每日写"。
> 修法:把"写日记"降级为静默维护任务(与 `episodic_decay` 同类),"发消息分享"保持原 gating 不动。

---

## 1. 现状盘点(已确认)

写日记的唯一两条路径,都被发言门控绑架:

| 路径 | 位置 | 被什么压住 |
|---|---|---|
| proposer | `core/scheduler/triggers/time_based.py:135` `propose_daily_journal` → `after_send=_write_inner_daily_journal`(同文件 ~937) | ①23:00–23:59 窗口(`now.hour < 23` 直接 return,过午夜 hour=0 也不成立);②`quiet_floor_elapsed`(23点还在聊天→废);③`requires_state=[QUIET]`;④`gating.py` 每日发送预算 `daily_budget_filtered`(~251);⑤与其他 proposal 拼 urgency(`picked_highest_urgency`,~256) |
| legacy | 同文件 `_check_daily_journal`(~546):先 `_pipeline_send` 再循环 `_generate_and_store_diary` | `legacy_tick_should_send()` + `_is_ready("daily_journal")` + 同样的 23 点窗口;且 send 抛异常时日记也不写(在同一个 try 里、send 之后) |

`_generate_and_store_diary(oid, char_id)`(同文件 456)本身与发言无关:读当日 event_log
→ 两次 LLM 调用(事件层+感受层)→ 写 `get_paths().yexuan_inner_diary(char_id)/{logical_day}.md`。
文件名已经用 `logical_day()`(`core/scheduler/rhythm.py:25`,cutoff=5点,凌晨算前一天)。

已知坑:`event_log.get_recent_days`(`core/memory/event_log.py:286`)按**日历日** `datetime.now()`
往回数,不认 logical day。若在午夜后补写,`days=1` 只会读到新日历日的空日志,漏掉正主。

---

## 2. 方案

### 2.1 新增静默维护检查 `_check_inner_diary_write()`

放 `core/scheduler/triggers/time_based.py`,模式照抄 `_check_episodic_decay`(不发言、不走
gating、不需要 `legacy_tick_should_send`):

```python
async def _check_inner_diary_write():
    """静默写各角色内心日记。与 daily_journal 主动发言完全解耦。"""
    now = datetime.now()
    # 窗口:23:00 起,跨午夜宽限到次日 05:00(LOGICAL_DAY_CUTOFF_HOUR)
    if not (now.hour >= 23 or now.hour < LOGICAL_DAY_CUTOFF_HOUR):
        return
    if not _is_ready("inner_diary_write"):
        return
    oid = _owner_id()
    if not oid:
        return
    for _cid in _diary_char_ids():
        # 幂等主闸:当日(logical day)文件已存在则跳过,不发 LLM 调用
        diary_file = get_paths().yexuan_inner_diary(char_id=_cid) / f"{logical_day().strftime('%Y-%m-%d')}.md"
        if diary_file.exists():
            continue
        try:
            await _generate_and_store_diary(oid, _cid)
        except Exception as e:
            log_error(f"scheduler._check_inner_diary_write[{_cid}]", e)
    _mark("inner_diary_write")
```

要点:

- **幂等靠文件存在性**,不靠 `_last_trigger`(重启后内存态丢失也不重复写)。全部角色文件已存在时同样 `_mark`,避免每 60s 空转扫描。
- `_COOLDOWNS`(`core/scheduler/loop.py:~44`)加 `"inner_diary_write": 2 * 3600`(冷却只是防抖,幂等由文件保证;2h 让午夜前失败的还有跨午夜重试机会)。
- 接入 `loop.py` 主循环 `asyncio.gather`(~945)与 `_trigger_names` 列表(~931),两处同步加,顺序对应。

### 2.2 修 `_generate_and_store_diary` 的跨午夜读日志

函数内改一行取数逻辑:

```python
days = 2 if datetime.now().hour < LOGICAL_DAY_CUTOFF_HOUR else 1
today_log = get_recent_days(oid, days=days)
```

(凌晨补写时把昨天日历日包含进来。多带的几小时新日志无害——日记写的本来就是"这一逻辑日"。)
注意 `get_recent_days` 目前签名不带 `char_id` 转发的问题不在本 brief 范围,保持现状调用。

### 2.3 摘除旧路径的写日记副作用

- `propose_daily_journal`(time_based.py:135):删除 `after_send=_write_inner_daily_journal`。`_write_inner_daily_journal`(~937)若无其他引用则整个删除。
- `_check_daily_journal`(~546):删除 `for _cid in _diary_char_ids(): _generate_and_store_diary(...)` 段。
- **发言部分一律不动**:`_pipeline_send("深夜,他回想起…")`、`_mark("daily_journal")`、gating 竞争、`loop.py` 783 附近的 defer 释放分支,全部保持原样。日记分享消息该被压就被压,那是预期行为。

### 2.4 文档同步

- `docs/scheduler.md`:daily_journal 条目改为"发言触发器,不再负责写日记";新增 `inner_diary_write` 静默任务条目(窗口、幂等、冷却)。
- `AGENTS.md` 关键文件速查无需改(无新文件)。

## 3. 测试

新增 `tests/test_inner_diary_write.py`(参考现有 scheduler trigger 测试的 monkeypatch 手法):

1. **解耦生效**:mock 时间 23:30、mock gating 全拒(或根本不跑 gating)→ 直接调 `_check_inner_diary_write()` → 日记文件生成(LLM 调用 mock 掉)。
2. **幂等**:文件已存在 → 再调不触发 LLM(断言 mock 的 llm_client.chat 未被调用)。
3. **跨午夜**:mock 时间 01:00 → 写入的是**前一逻辑日**文件名;`get_recent_days` 收到 `days=2`。
4. **窗口外不跑**:mock 时间 14:00 → 不写、不 mark。
5. **回归**:`propose_daily_journal` 返回的 proposal 无 `after_send` 写日记副作用(执行 execute(dry_run=True) 后无日记文件)。

---

## 4. 不做什么

- 不改 gating / 预算 / urgency 任何逻辑。
- 不改 `_generate_and_store_diary` 的 prompt 与双层结构。
- 不处理 `get_recent_days` 的 logical-day 语义重构(只在调用侧用 days=2 绕过)。
- 不动 `diary_reminder` / `diary_inject` / `diary_share_reminder`(那三个是用户日记,不是角色日记)。
