# CC 任务 19：主动触发底层修复 — 统一发送预算 + 召回锚点治理

> 背景：`docs/proactive-trigger-audit.md`（Brief 08 #5）做了止血（全局最小间隔调大到 35100s），但茶茶反馈**根本无效**。本次由 fable 实读代码 + 实测生产数据定位根因。结论：止血参数**从未在运行中的进程里生效**，且频率大头根本不走全局间隔那条路。
> 本单是架构级修复，按 A → B → C → D 顺序施工，A 可独立先上。

---

## 〇、实测证据（2026-07-02，data/ 生产数据）

近 11 小时内 owner 实际收到的主动消息（`history.json` 的 `[触发: *]` stub 时间线）：

| 距今 | 触发器 | 距上一条 |
|---|---|---|
| 10.46h | sensor_aware | — |
| 10.20h | sensor_aware | 16min |
| 10.14h | weather_alert | **4min** |
| 9.15h | random_message | 59min |
| 8.37h | spontaneous_recall | **47min** |
| 6.91h / 6.23h / 5.91h | sensor_aware ×3 | 19~40min |

**11 条主动消息 / 11 小时，配置的全局间隔是 9.75 小时。** 其中 7 条来自 sensor_aware（完全不走 gating）；走 gating 的 weather_alert → random_message → spontaneous_recall 间隔 47~59 分钟——与 config.yaml 里的 35100s 完全不符，但与**旧默认值 2700s（45min）±20% jitter 精确吻合**。

另：`gating_shadow.jsonl.1.gz` 120 小时内 **3269 次 picked_highest_urgency**，其中 letter_writer 连续数小时每 tick（60s）被选中一次——每次 pick 都是一整条 LLM 调用。

## 一、根因清单（按危害排序）

### RC1 · sensor_aware 完全旁路（主频率源）

本机 config `scheduler.sensor_aware.enabled: true`。`_check_sensor_aware()` → `handle_tick()`（`triggers/sensor_aware.py`）**不经过 `gating._decide()`**：无状态机、无 active-window、无 DND、无全局间隔，只有自己 8 分钟的 `_PROACTIVE_COOLDOWN_SECS`。发送后也**不调用 `_mark_global_proactive()`**——所以它既不受约束、也不让别人知道它刚说过话（10.14h 那条 weather_alert 在 sensor_aware 发言 4 分钟后跟进，就是这个原因）。

### RC2 · config 单例缓存 → 手改 config.yaml 对运行中进程无效

`core/config_loader.py` 的 `get_config()` 是**永久缓存单例**，只有 `reload_config()`（仅 `PUT /scheduler/config` 调用）或重启才重读磁盘。止血时是**手改 config.yaml** 把 gap 调到 35100——运行中的进程根本没加载它，一直用启动时的旧值（实测行为对应 2700s）。这就是"止血根本无效"的直接答案。

### RC3 · 全局间隔 jitter 每次调用重抽

`_global_proactive_gap_ready()` 里 `jitter = gap * 0.2 * (random()*2-1)` 是**每 tick 重新抽的**。调度器每 60s 问一次，等价于"反复抽签直到抽到最松的"——有效间隔恒等于 **0.8 × 配置值**，且抖动完全失去"错峰"意义。

### RC4 · 发送失败不留痕 → winner 每 tick 全量 LLM 重试

`execute_prompt()` 只在 `sent=True` 时 `_mark()`。当 `_pipeline_send` 返回 None（perceive_event DUPLICATE / Dream Guard / LLM 空回复）或 letter_writer 内部质量门不过时，**没有任何退避**——下个 tick 同一 proposer 重新报名、重新胜出、重新跑一遍完整 pipeline。letter_writer 的 3269 次 pick 就是这么烧出来的（它内部先跑 LLM 写信、再质量门拒发，每 60 秒一遍）。

### RC5 · 全局记账只覆盖 execute_prompt 一条路

`_mark_global_proactive()` 只在 `execution.execute_prompt()` 成功后调用。**不记账的发言出口**：sensor_aware（RC1）、`desktop_wake` Path B（每次开客户端现场生成问候，只有 60s perceive dedup）、`manual_trigger`（管理面板，测试用，绕过属设计但也该记账）。gating 看到的"上次主动时间"是残缺的。

### RC6 ·（乱召回根因）search_query 锚点宽泛或直接拿 prompt 全文

`fetch_context(oid, search_query or prompt)`——主动触发的 RAG/event_log/episodic 召回锚点是触发器种子词，不是真实用户输入。实测：festival/timenode/daily_journal 等用 `"今天"`，diary 系用 `"日记"`，而 **sensor_aware / presence_nag / dream_exit / overflow 等不传 search_query，直接用整段括号叙事 prompt 当检索词**。宽泛词命中一堆无关记忆 → 注入 prompt → 角色"胡乱召回然后说一大堆废话"。

### 附 · 已排查、非问题

- 前端（Emerald-client）：`desktopWake` 有 localStorage 去抖，每窗口会话至多一次；`dream_invite`/`toy_invite` 只是 WS listener，后端无发射点（前端预埋，非当前风险）；客户端不自产发言。安卓端本单不动。
- `execute_dryrun.jsonl` 里成批的 blocked=True（reminders/morning_greeting/random_message 同时刻）与时间上和手动测试会话吻合，判定为管理面板 manual_trigger 测试痕迹，不是自动路径；若施工中发现自动来源再单独开单。

## 二、施工项

### A · 止血修复（先做，半天，每项独立可验）

**A1 config 热加载**：`get_config()` 增加 mtime 检查——每次调用 `stat()` config.yaml，mtime 变了自动 `reload_config()`（stat 开销可忽略；若担心高频路径，可只在 `loop._cfg()` 侧做，每 tick 至多一次）。同时：调度器启动时和检测到 gap 变更时 INFO 打一行 `[scheduler] effective global_proactive_min_gap_seconds=XXXXX`，让"生效没生效"直接可见。

**A2 jitter 一次性采样 + 只增不减**：废弃"每次检查重抽"。改为 `_mark_global_proactive()` 时计算并持久化 `next_allowed_ts = now + gap + uniform(0, 0.2*gap)`（jitter 只加不减），`_global_proactive_gap_ready()` 退化为 `now >= next_allowed_ts`。落盘进 `scheduler_cooldowns.json`（新 key，如 `__next_proactive_ts__`），重启不丢。

**A3 sensor_aware 纳管**：`handle_tick()` 在 plan() 通过后、跑 LLM 前，增加三道检查：`_global_proactive_gap_ready()`、`is_dnd(uid)`、（可选）TriggerState==QUIET；任一不过则记 audit `stage="global_gap_blocked"` 等并返回。发送成功后调用 `_mark_global_proactive()`。8 分钟私有冷却保留但仅作下限。**注意顺序：检查放在 judge/LLM 之前，省掉被拦时的整条 pipeline 开销。**

**A4 失败退避**：`execute_prompt()` 在 `sent=False` 时对该 trigger 打 attempt-cooldown（内存 dict 即可）：首次 15min，指数翻倍，封顶该触发器自身 `_COOLDOWNS`；proposer 冷却检查（`_is_ready`）同时看正式冷却和 attempt-cooldown。letter_writer 内部质量门拒发同样走这条（它的 execute 返回 sent=False 已覆盖）。验收标准：`execute_dryrun.jsonl` 不再出现同 trigger 连续每 tick 的条目。

### B · ProactiveLedger：统一发送预算与记账（1~2 天）

新文件 `core/scheduler/proactive_ledger.py`，成为**所有主动发言的最后一道闸 + 唯一记账点**（决策权威仍是 `gating._decide()`，ledger 是它查询的数据源，两层不冲突）：

```
can_send(trigger_name, *, priority) -> (bool, reason)
    # 检查 next_allowed_ts（A2）+ 当日预算；emergency 恒 True 但仍记账
record_send(trigger_name, *, channel) -> None
    # 写 next_allowed_ts、当日计数、最近 gist（接管 _append_proactive_recent）
snapshot() -> dict   # 观测用
```

- **接入点（全量）**：`execution.execute_prompt()`（替换 `_mark_global_proactive`）、`sensor_aware.handle_tick()`（替换 A3 的直调）、`desktop_wake` Path B（**record-only**：wake 问候语义上必须发，但要计入账本，防止刚 wake 又来一条 random_message）、`manual_trigger`（record-only）、watch emergency（exempt 但 record）。
- **当日预算**：`scheduler.max_daily_proactive`（默认 8，emergency 豁免），按 `rhythm.py` 逻辑日重置。这是比"最小间隔"更符合直觉的总闸：间隔管"别连珠炮"，预算管"一天别太吵"。
- 持久化沿用 `scheduler_cooldowns.json` 或独立 `data/runtime/proactive_ledger.json`（原子写，`core/safe_write.py`）。
- **观测端点** `GET /scheduler/proactive-ledger`：返回 effective gap（当前内存值）、next_allowed_ts、今日已发条数/预算、最近 3 条 gist。管理面板加一小块展示。以后"止血生效没有"一眼可查。
- 60s TTL perceive dedup 保留（防并发兜底），但不再是跨触发器限流的承担者。

### C · 召回锚点治理（1 天，解决"乱召回说废话"）

给 `execute_prompt()` / `_pipeline_send()` 增加 `recall_policy` 参数，三档：

| 档 | 行为 | 适用 |
|---|---|---|
| `"none"`（**主动触发默认**） | `fetch_context` 跳过 episodic/event_search/web_recall 检索层，只保留 identity/mood/short_term/花园等状态层 | random_message、weather_alert、sensor_aware、presence_nag、garden_*、festival、timenode、daily_journal¹、morning/night、diary 系 |
| `"anchored"` | 用触发器自带的具体锚点（话题 key、被选中记忆的原文）检索 | topic_followup、spontaneous_recall（time_based.py:817 已传具体 query，保持）、birthday（"生日"）、period（"生理期"） |
| `"seed"` | 现状（用 prompt 全文），仅兼容过渡，目标是没有触发器用它 | — |

¹ daily_journal 改为只注入当日 event_log 原文（它 prompt 里本来就带 log_hint），不做语义检索。

理由：主动开口的"由头"已经在种子 prompt 里写死了（天气、想她了、看到她久坐），角色不需要再被"今天"这种词捞出来的三条旧情景记忆带偏。想让主动消息引用记忆的触发器（followup/recall）本来就有精确锚点。

同步：`docs/scheduler.md`「search_query 召回锚点说明」和 `docs/prompt-layers.md` 相应更新；prompt_capture 的 `origin.search_query` 加记 `recall_policy` 字段。

### D · 决策记录 + 清理（半天）

1. **不建真发送队列**——正式记入 `DESIGN.md` 或 `docs/scheduler.md`：单用户场景下"每 tick 单 winner + ledger 间隔/预算 + defer_queue"已提供串行化与限速；真队列引入合并/排序/过期三套新语义，收益不成比例。审计 §五.1 的担忧（相邻 tick 背靠背双发）由 A2 的 next_allowed_ts 硬性解决。
2. **平台级限流不做**——desktop/QQ 共享 uid 级账本是刻意设计（防同人双份），维持；presence_nag 独立 fanout desktop 维持。审计 §五.2 关闭。
3. **dream_invite / toy_invite**——后端无发射点、前端 listener 预埋，审计 §五.3 关闭（标注"规划中"）。
4. 删除 garden_daily.py / garden_water.py 里被 `EXECUTE_MODE` 挡死的 legacy `_emit` for-loop（审计 §五.4）。
5. `admin/routers/scheduler.py` PUT 的 `(0,24]` 小时上限维持；GET 返回体加 `effective_gap_seconds`（内存实际值）与文件值并列，两者不一致即提示需 reload（A1 落地后应恒一致）。

## 三、验收

1. **A1**：运行中手改 config.yaml 的 gap，≤60s 内日志出现新 effective 值，无需重启。
2. **A2**：单测——mark 后在 `[gap, 1.2*gap]` 内任意时刻 `gap_ready()` 恒 False，之后恒 True；重启后 next_allowed_ts 从文件恢复。
3. **A3/B**：sensor_aware 发言后，紧随其后的 gating tick 里所有非 emergency 候选 reason=`global_gap_filtered`（看 gating_shadow.jsonl）；反向同理。
4. **A4**：制造一个必拒发的触发（如 letter_writer 质量门），观察 execute_dryrun.jsonl 中该 trigger 出现频率呈 15/30/60min 退避而非每 tick。
5. **B**：一天内非 emergency 主动消息条数 ≤ `max_daily_proactive`；`GET /scheduler/proactive-ledger` 数字与实际一致。
6. **C**：Prompt 层检视页查看 random_message / sensor_aware 主动轮，episodic/event_search 层为空；topic_followup 轮 rag_query 为话题原文而非"今天"。
7. pytest 补 ledger 与退避单测；`python tests/run_eval.py` 不受影响（未动 tag_rules）。

## 四、风险与边界

- A3 把检查放到 judge 之前会让 sensor_aware audit 的 `judge_*` 字段在被拦 tick 为 null——audit 结构保持完整（现状约定字段拿不到为 null），前端无需改。
- desktop_wake record-only 意味着"开机问候后 9.75h 内没有其他主动消息"——这正是期望行为；若茶茶觉得太安静，调 gap 或预算即可（现在终于调得动了）。
- 全部改动限后端仓库；Emerald-client 无需配合改动。
- 改 `core/scheduler/*` 后按 doc-sync hook 要求同步 `docs/scheduler.md`（全局间隔、sensor_aware、新增 ledger 章节）。
