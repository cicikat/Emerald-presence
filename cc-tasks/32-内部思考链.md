# Brief 32 · 内部思考链(原生 reasoning + 前置独白,可开关)

> 配对文档:`Emerald-client/cc-tasks/17-思考开关设置.md`(前端)。
> 目标:主生成前有一段**不输出给用户**的内心思考,缓解"不带思考有些生硬"。
> 两条路按 preset 能力自动选:支持原生 reasoning 的模型直接开;不支持的走前置独白
> (一次轻量调用产出内心活动,注入主调用后丢弃)。全局开关,默认关。
>
> 铁律:思考内容**永不**进 short_term history、永不广播、永不落 event_log。
> 唯一合法去向是当轮主调用的 messages,用完即弃。

---

## 1. 现状盘点(已确认)

| 事实 | 位置 |
|---|---|
| preset 参数合并有 provider_kind **白名单过滤**(anthropic_compat 只放行 temperature/top_p/max_tokens)→ reasoning 类参数会被白名单剥掉,不能走 params 通道 | `core/model_registry.py` 参数合并;docs/model-presets.md 适配表 |
| 非流式主调用 `chat()`、流式 `chat_stream()`(仅无工具主生成)、loop 用 `chat_turn()`(Brief 28) | `core/llm_client.py` |
| 现实侧输出清洗已有独立模块 | `core/reality_output_scrubber.py` / `core/reality_output_guard.py` |
| 新 prompt 层必须带 `_layer`(硬性规则3);层表在 prompt_builder ~1635 | `core/prompt_builder.py` |
| settings 热生效模式先例 | `admin/routers/settings_screen_peek.py` / `settings_tool_loop.py`(Brief 28) |

## 2. 配置

```yaml
thinking:
  enabled: false          # 总开关,默认关
  mode: auto              # auto | native | monologue
  monologue_max_tokens: 200
  apply_to_proactive: false   # scheduler 主动消息是否也思考(默认省钱不开)
```

preset 侧新增可选字段(`model_presets.presets.<name>`):

```yaml
    deepseek-reasoner:
      provider_kind: deepseek
      model: deepseek-reasoner
      reasoning_native: true          # 声明该 preset 有原生思考
      reasoning_extra_body: {}        # 原样经 OpenAI client 的 extra_body 透传,绕过参数白名单
```

`mode: auto` 语义:当前 chat preset `reasoning_native: true` → native;否则 → monologue。

## 3. native 路线

- `reasoning_extra_body` 在 `chat/chat_stream/chat_turn` 构建请求时并入
  `extra_body=`(OpenAI python client 支持),**故意绕过白名单**——这是逃生舱,
  网关方言差异(DS 的 reasoner 无参、o系的 reasoning_effort、anthropic 网关的
  thinking budget)全靠用户在 preset 里自己填,代码不做每家适配(Brief 29 §4.3 同原则)。
- **剥离思考内容,三道防线**:
  1. 非流式:响应 message 若带 `reasoning_content` 字段 → 直接忽略,只取 `content`。
  2. 内联 `<think>...</think>`(部分网关/本地模型会内联):`llm_client` 出口统一正则剥除
     (含跨行,`re.S`),连同变体 `<thinking>`。
  3. `reality_output_scrubber` 追加同模式兜底(防其他路径漏网)。
- **流式**:`chat_stream` 跳过 `delta.reasoning_content`;对内联 think 流做缓冲——
  首个 chunk 以 `<think` 开头时进入缓冲态,直到读到 `</think>` 才开始对外 yield。
  缓冲态超 60s 或流结束仍未闭合 → 剥掉已缓冲的 think 前缀后把剩余部分放行(fail-open)。
- **与 tool loop 的交互**:reasoner 类模型多数不支持 function calling(如 deepseek-reasoner),
  此时 `tool_call_mode` 自然不满足 loop 激活条件,互斥天然成立,无需代码;
  两者都支持的模型(claude 系)native 思考 + loop 可叠加,每步都思考——成本翻倍,
  docs 里写明白让用户自己权衡。

## 4. monologue 路线(前置独白)

- 主生成前一次轻量调用:`call_category="monologue"`,routing_profiles 各 profile 补该
  key(缺省回落 chat 的既有解析规则会导致独白走主力贵模型——所以 **profile 里显式指到
  便宜 preset**,config.example 示范)。
- prompt(简版,cc 按现有二人称口径微调):

```
你是{char_name}。看到她刚说的话,先在心里想一下:她想要什么/你此刻的情绪/你打算怎么回。
口语化、跳跃、不成段都行,≤120字。只输出内心活动本身。
输入:她的话 + 你的心情({mood_hint})+ 最近两轮对话摘要
```

- 注入:作为 system 消息插在 messages 尾部、用户消息之前,
  `_layer: "11.7_inner_monologue"`(登记层表),文案框架:
  `（你此刻的内心活动,不要直接复述:{monologue}）`。
- 独白调用失败/超时(10s)→ 跳过注入,主生成照常,fail-open,log 一条。
- 与 loop 叠加:loop 激活轮独白只做一次(首步前),不逐步做。
- char_id 穿线(Brief 30):独白调用传当前说话角色的 char_id。

- docs/model-presets.md:`reasoning_native`/`reasoning_extra_body` 字段 + extra_body 绕白名单的逃生舱语义。
- docs/prompt-layers.md:`11.7_inner_monologue` 层。
- `tests/test_thinking.py`:
  1. 开关关 → 主调用 messages 与现状逐字节一致(回归)。
  2. monologue:注入位置正确、带 `_layer`、独白失败时静默跳过、≤max_tokens 传参正确。
  3. native 非流式:reasoning_content 被忽略;内联 `<think>` 被剥(含跨行/变体标签)。
  4. native 流式:缓冲态直到闭合才 yield;未闭合流 fail-open 放行;delta.reasoning_content 跳过。
  5. extra_body:reasoning_extra_body 出现在请求 kwargs,且**不**经白名单过滤;普通 params 白名单行为不变。
  6. auto 模式判定:reasoning_native true/false 分别走对路线。
  7. scrubber 兜底:构造带 think 标签的 assistant 文本过 scrubber → 干净。
  8. history 铁律:mock 全链跑一轮,断言 short_term 写入内容不含独白/思考片段。

## 7. 风险与不做什么

- **延迟**:monologue +1 次调用;native 首 token 显著变慢(缓冲)。默认关 + 前端可随手关。
- **独白质量**:小模型独白可能平庸甚至带偏主生成——观察期建议只在 claude/gpt preset 上开;
  独白被 prompt 注入攻击的风险与其他注入层同级,接受现状。
- 不做:per-char thinking 开关(需要了再加 presence_ext 字段,机制现成);思考内容展示给
  用户(哪怕折叠 UI 也不做,防止用户对着思考调教出双层人格);把思考写入任何记忆层。
